Stack Docs

Introduction

Stack is a lightweight framework for ROBLOX, which makes development a lot smoother by providing clean and simple communication, while keeping the code organized.

Installation

Download by clicking the link below.

Direct Download

When the download finished, open Roblox Studio and either create a new experience or open an existing experience.

  1. Drag Stack.rbxm into the game it should appear in Workspace if done correctly.
  2. Then drag Stack into ReplicatedStorage and open it
    • Drag Client into StarterPlayer > StarterPlayerScripts
    • Drag Server into ServerScriptStorage

Congratulations, you successfully imported and set up Stack you are ready to go!

Preface

For the sake of simplicity, we will use the Client side of Stack to do the explaination, everything that is mentioned here below will be the same for the Server. The only difference being its location of where it runs. However, there are still some notable differences between the two. But we will get into that in the Examples

Client

During the installation, you dragged a folder called Client into StarterPlayerScripts. If you look inside of the folder. You will see a Main script, lets break it down real quickly.

Breakdown of main

shared.Stack = require(script.Client)

This will create a global to require the Client module, which handles everything needed for the client side of Stack. The reason we are setting a global is for the sake of easy access to the module.

shared.Stack.run {
    Location = script.Parent.ClientModules,
    Excluded = {}
}

This is where we call the run function, as you can see it takes in a table value where you have a Location and Excluded option. Lets break them down

  • Location: This is where you will load your modules from, by default you shouldn't really need to change this. Cause its located directly in the Client folder. Although do keep in mind this option uses GetDescendants to fetch every ModuleScript.
  • Excluded: This option takes a table and its used to exclude any module you want (preventing it from loading). If we have a module named SomeModule then to exclude it we would add it as a string inside the table like so "SomeModule"

Creating a module script

Lets start by creating a new ModuleScript instance inside of ClientModules folder, lets name it TestClient and insert the following contents inside of it.

local Stack = shared.Stack
local TestClient = Stack.Create { script }

function TestClient.main()
    print("Hello from, TestClient!") 
end

return TestClient

Breakdown of TestClient

local Stack = shared.Stack

Sets our variable to the global that was originally declared inside of our Main script

local TestClient = Stack.Create { script }

Our variable here is the name of the module itself, although this name can be anything its a good practice to keep this variable to the name of the module, Create is a function that takes table argument. this function is what adds the module to Stack, it should only be called once. Inside this table the very first thing you should always put inside is script

function TestClient.main()
    print("Hello from, TestClient!")
end

Here we have a special function called main and it will run when the module finished loading. However this function is not always necessary, only use it when you need to run something from the module.

If you play test the game now, you will see in the output. Hello from, TestClient!

More on main function

One thing to keep in mind, main actually gets called as a method, so if you do :main() you will get access to self

function MyCoolModule:main()
    print(self) -- MyCoolModule 
end

Communication

One of the powerful features of Stack is its ability to communicate modules between each other. This allows for much more organized and flexible code.

Basic Example

local Stack = shared.Stack
local Addition = Stack.Create { script }
Addition.FavoriteNumbers = {1, 2, 3}

local function Subtraction(x, y)
    return x - y
end

function Addition.add(x, y)
    return x + y
end

return Addition

local Stack = shared.Stack
local SomeOtherModule = Stack.Create { script }
local Addition = Stack.Import("Addition")

function SomeOtherModule.main()
    print(Addition.add(2, 2))
    print(Addition.FavoriteNumbers)
end

return SomeOtherModule

Output: 4
{1, 2, 3}

Here we have two modules Addition and SomeOtherModule, you can see inside of Addition we have a function add and also FavoriteNumbers and it gets passed into Addition but if you notice Subtraction does not get passed, meaning we can not call it from other modules.

Now if we take a look inside of SomeOtherModule we will notice we use Import and inside of main we print out some of the things passed from addition

Breakdown of Import

-- Example 1
local SomeModule = Stack.Import("SomeModule")
SomeModule.SomeFunction()

-- Example 2
local Import = Stack.Import {
    "SomeModule", 
    "AnotherModule",
}

Import.SomeModule.SomeFunction()
Import.AnotherModule.AnotherFunction()

Import is diverse and you can use it for single modules or multiple depending on your preference.

NOTE: Import does yield until module(s) are fully loaded, so its recommended to keep it under the Create function

Bidirectional Example.. oh uh

local Stack = shared.Stack
local Module1 = Stack.Create { script }
local ImportModule2 = Stack.Import("Module2")

function Module1.SomethingFromModule1()
    print("Hello from Module 1")
end

function Module1.main()
  ImportModule2.SomethingFromModule2()
end

return Module1

local Stack = shared.Stack
local Module2 = Stack.Create { script }
local ImportModule1 = Stack.Import("Module1")

function Module2.SomethingFromModule2()
    print("Hello from Module 2")
end

function Module2.main()
    ImportModule1.SomethingFromModule1()
end

return Module2

Remember previously it was mentioned that Import will yield, and if two modules try to import each other this way it will yield infinitely.

While it's generally not recommended to have two scripts import each other, there may be situations where this is necessary. One solution is to use main to ensure that a module is fully loaded before running main. This ensures that main is only run after the loading process is complete, similar to the initialization of the module(s).

The Solution!

local Stack = shared.Stack
local Module1 = Stack.Create { script }

function Module1.SomethingFromModule1()
	print("Hello from Module 1")
end

function Module1.main()
	Stack.Import("Module2").SomethingFromModule2()
end

return Module1

local Stack = shared.Stack
local Module2 = Stack.Create { script }

function Module2.SomethingFromModule2()
	print("Hello from Module 2")
end

function Module2.main()
	Stack.Import("Module1").SomethingFromModule1()
end

return Module2

Output: Hello from Module 1
Hello from Module 2

Now you can see the code runs perfectly fine, although if you want you could set it as a variable from main

local Stack = shared.Stack
local Module2 = Stack.Create { script }
local IMPORTED_Module1

function Module2.SomethingFromModule2()
	print("Hello from Module 2")
end

function Module2.main()
	IMPORTED_Module1 = Stack.Import("Module1")
end

return Module2

If you do pass/include IMPORTED_Module1 somewhere else and attempt to use it before assigned inside of main then it will throw an error. So its not recommended to do this

What about race conditions?

Stack yields when importing modules once. The Client waits for the Server to finish loading and executing, including main, to prevent communication before readiness and players joining faster than the Server can run everything needed.

Networking

Stack has the ability to network using ROBLOX's built-in Remote Functions and Events, but this process is simplified and completed entirely within the code.

Server

local Stack = shared.Stack
local TestServer = Stack.Create { script }

TestServer.Network.TestRemote = {"Remote", 1}
TestServer.Network.TestFunction = {"Function", 1}

function TestServer.main()
	TestServer.Network.TestFunction:Connect(function(Player, Key)
		if Key == "Secret" then
			return "You got the secret!"
		else
			return "No secret for you!"
		end
	end)

    game:GetService("Players").PlayerAdded:Connect(function(Player)
		TestServer.Network.TestRemote:Fire(Player, "Hey from the server!")
	end)
end

return TestServer

Client

local Stack = shared.Stack
local TestClient = Stack.Create { script }
local Network = Stack.Network

function TestClient.main()
	Network.TestRemote:Fire("said hello")
	print(Network.TestFunction:Invoke("Secret"))
    task.wait(1)
	print(Network.TestFunction:Invoke("SomethingElse"))
end

return TestClient

Breakdown of Server

TestServer.Network.TestRemote = {"Remote", 1}
TestServer.Network.TestFunction = {"Function", 1}

Behind the scenes Server creates an empty table also known as Network. This table is apart of the initalization where it constructs Remote Functions and Events.

  • Remote = RemoteEvent
  • Function = RemoteFunction

The number next to "Remote" and "Function" indicates the cooldown time. If no number is included, the default cooldown is set to 1/60, which is ROBLOX's maximum framerate. To remove the cooldown, use 0.

Why a cooldown?

ROBLOX does not offer a built-in method for developers to implement cooldowns, meaning remotes can be fired without limit. This can result in undesired outcomes and potential harm to the game.

Although not the only factor at play, there are limits to the number of calls a Player can make. By examining the Network module within Server, the following code can be found at the top:

local function Template()
    return {
        RemoteCooldowns = {},
        Timeout = {
            AllowTimeout = false,
            LastCall = os.clock(),
            MaxCalls = 200,
            Calls = 0
        }
    }
end
  • If you desire to disable this feature set AllowTimeout to true.
  • To increase the limit, modify the MaxCalls value to your desired amount.

If you want to change the functionality of what happens to the Player if limit is exceeded. You can change this code:

if Session.Timeout.Calls > Session.Timeout.MaxCalls then
    Player:Kick("Send fewer requests.")
end

Both cooldown and call limit only applies to the Client

Establishing a connection to the Remotes

If you look inside of main you can see we have these following lines of code

TestServer.Network.TestRemote:Connect(function(Player, Message)
    print(Player, Message)
end)
	
TestServer.Network.TestFunction:Connect(function(Player, Key)
    if Key == "Secret" then
        return "You got the secret!"
    else
        return "No secret for you!"
    end
end)

As mentioned in Communicating everything is initialized before main can execute. This is why we use Connect inside of main. If mentioned outside of main you will recieve an error.

Breakdown of Client

local Network = Stack.Network

Inside of our Client at the top we have access to Network, which basically just retrieves all the created Remotes and stores it inside of a table that we can use later in our code.

Network.TestRemote:Fire("said hello")
print(Network.TestFunction:Invoke("Secret"))
task.wait(1)
print(Network.TestFunction:Invoke("SomethingElse"))

Inside of our main we have a few lines of code that basically communicates with the server.

  • When dealing with RemoteEvents use Fire
  • When dealing with RemoteFunctions use Invoke

When running this code you should see this in the output

Output: <PLAYER> said hello
Secret
SomethingElse

Since when we created the Remotes on the server with a cooldown of 1 second. We used task.wait(1) to wait 1 second before Invoking TestFunction again. If we were to run this example again but instead task.wait(1) was lesser than 1 we would simply recieve nil as our result. As for events if you execute Fire while under cooldown then it will simply do nothing.

Server to Client

A simple example of how we can make the Server fire a remote and our Client listens.

Server

function TestServer.main()
	game:GetService("Players").PlayerAdded:Connect(function(Player)
		TestServer.Network.TestRemote:Fire(Player, "thanks for joining!")
	end)
end

Client

function TestClient.main()
    Network.TestRemote:Connect(function(ServerMessage)
        print(ServerMessage)
    end)
end

The Server uses PlayerAdded to wait for a player to join, then it will Fire the event we created before.

The Client will listen for the call from the Server, when it recieves it the Client will print out the argument.

What if the player connects before the server runs?

As mentioned here is that the Client will wait on the server, so if the Client needs to tell the Server it's also fully loaded. You could create a Remote that the Server can listen to and handle your logic from there.

Packages

If you look in ReplicatedStorage inside the Stack folder, you will see Packages, this is shared between the Client and Server. They just act as regular modules.

Packages uses GetDescendants to require all ModuleScript instances under it. Stack comes with two modules by default.

  • Promise
  • Cooldown, which is a basic cooldown module with one function.
local Stack = shared.Stack
local TestClient = Stack.Create { script }
local Network = Stack.Network
local Packages = Stack.Packages

function TestClient.main()
	Packages.Promise.new(function(resolve)
		if 2 == 2 then
			resolve("two is two")
		end
	end):andThen(function(result)
		print(result)
	end)
end

return TestClient

Output: two is two

You can see here in the example Packages operates basically the same way import does when using tables.
